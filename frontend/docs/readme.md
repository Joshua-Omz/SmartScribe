
Frontend Integration Contract (Source of Truth)

Project: SmartScribe

This document serves as the absolute source of truth for the interaction between the Web/Mobile Frontend and the Go Backend API. If the frontend adheres strictly to this contract, integration will be seamless.

1. Base URL & Environment

Local Development Base URL: http://localhost:8080

Note: The backend is configured to handle CORS for local development, allowing requests from different local ports (e.g., if the frontend is running on a live-server at port 5500).

2. The Primary Endpoint: Dictation Submission

Request Specifications

Endpoint: POST /api/transcribe

Content-Type: multipart/form-data

Maximum Payload Size: Strictly enforced at 10MB. Requests exceeding this will be instantly terminated by the backend (413 Payload Too Large).

Expected Payload Structure

The backend expects the audio file to be sent inside a FormData object. The field name must be exactly audio.

Frontend Implementation Example (JavaScript):
According to MDN Web Docs for FormData and fetch:

// Assume 'audioBlob' is the Blob generated by the MediaRecorder API
const formData = new FormData();

// The field name "audio" is MANDATORY. 
// "dictation.webm" is a suggested filename, but the backend doesn't strictly validate the name, only the binary content.
formData.append("audio", audioBlob, "dictation.webm");

try {
    const response = await fetch("http://localhost:8080/api/transcribe", {
        method: "POST",
        body: formData, 
        // DO NOT explicitly set the Content-Type header. 
        // The browser automatically sets it to multipart/form-data with the correct boundary when passing a FormData object.
    });
    
    const result = await response.json();
    console.log(result);
} catch (error) {
    console.error("Network or parsing error:", error);
}


3. Response Specifications

The backend will always return JSON.

Success Response (HTTP 200 OK)

When the AI successfully transcribes and structures the audio.

{
  "status": "success",
  "raw_text": "Patient presents with a mild fever of 101 degrees and reports a headache. Prescribed paracetamol.",
  "structured_data": {
    "subjective": "Reports headache.",
    "objective": "Mild fever of 101 degrees.",
    "assessment": "Fever and headache.",
    "plan": "Prescribed paracetamol."
  }
}


Error Responses

The frontend must be prepared to handle the following HTTP status codes and display appropriate feedback to the user.

HTTP 400 Bad Request: Sent if the payload is missing the "audio" field or the multipart form is malformed.

HTTP 405 Method Not Allowed: Sent if the frontend attempts a GET request instead of a POST.

HTTP 413 Payload Too Large: Sent if the audio file exceeds 10MB. (Frontend Action: Notify the user to keep recordings under 5 minutes).

HTTP 500 Internal Server Error: Sent if the external AI service fails or the Go server encounters an unexpected issue.

Standard Error JSON Structure:

{
  "status": "error",
  "raw_text": "",
  "structured_data": null,
  "error": "Detailed error message here (e.g., 'Failed to contact AI service')"
}


4. Implementation Notes for the Frontend Dev

Audio Format: The backend is currently agnostic to the initial web audio format (e.g., .webm, .ogg). Send the blob exactly as the browser's MediaRecorder generates it. The backend OS script handles the necessary conversions.

Loading State: The transcription process involves calling external AI APIs. The frontend must implement a robust loading state (spinner/overlay) that remains active until the backend returns the 200 OK or an error.
